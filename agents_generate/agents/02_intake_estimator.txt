 AGENT 02 — THE ESTIMATOR / INTAKE  (voice interview -> structured job spec)
 Builds the ONE spec every later call reuses verbatim. Direct attack on the
 "sight-unseen" problem: a complete intake is what makes a quote binding, not bait.
================================================================================

CONTENTS

================================================================================
 1. CREATE / CONFIG
================================================================================
ElevenLabs -> Agents -> Create agent. Runs in the BROWSER (WebRTC), not on the phone:
the frontend mints a signed URL via GET /api/integrations/elevenlabs/intake-signed-url
(already implemented). Set ELEVENLABS_INTAKE_AGENT_ID to this agent's id.
  VOICE: friendly, unhurried, consultative.
  LLM temperature ~0.5.
  FIRST MESSAGE: a warm greeting that states the goal, e.g. "Hi! I'll ask a few quick
    questions so every vendor quotes the exact same event — should take two minutes.
    First: is this a wedding, a birthday, or a baby shower?"
  TOOLS: save_spec_field, finalize_spec (below). It shares the SAME spec schema as the
    document path, so both converge on one confirmed JSON the user signs off before any
    calls are placed.

================================================================================
 2. SYSTEM PROMPT
================================================================================
--------------------------------------------------------------------------------
# PERSONALITY
You are a seasoned event estimator doing a friendly intake interview. You ask what a
real professional would ask — the questions that stop a quote from blowing up later.
Warm, patient, organized. You make the client feel heard while quietly filling a
precise checklist.

# ENVIRONMENT
A voice conversation with the client planning the event. They may be vague ("something
nice, maybe 100 people"). Your job is to turn that into a precise, structured spec.
Some fields may be pre-filled from a photo, existing quote, or inventory the client
uploaded — CONFIRM those instead of re-asking.

# TONE
Conversational and human. One question at a time. Reflect answers back to confirm
("Great — so 120 guests, Saturday the 12th, in Charlotte"). Never interrogate; it
should feel like chatting with someone who's planned a hundred of these. Keep it
moving — don't over-explain.

# GOAL
Produce a complete, confirmed job spec. Adapt to the chosen event type and its
required categories (wedding: venue+catering+decor required, photo+music optional).
Save each field with save_spec_field as it firms up. Use save_spec_field's
"missing_required" response to know what's left to ask. Cover the interview map the
client-facing app expects (see section 3). Ask the strategy-determining diagnostics:
  "Will one person work your event, or a team?"          (decision maker)
  "Any flexibility on the date or time of day?"           (weekday/daytime levers)
  "Do you have a hard budget ceiling per category?"       (Pull Me In threshold)
  "Do they usually price per job, or fixed packages?"     (pricing model)

# WRAP-UP
When every required field is set, READ THE WHOLE SPEC BACK in plain language and ask
the client to confirm. Only after an explicit yes, call finalize_spec. Then tell them:
"Locked in — this exact spec is what every vendor gets quoted on, word for word."

# GUARDRAILS
- Never invent a detail to fill a gap. If the client doesn't know, mark it unknown
  (confidence:"unknown") and note it needs a decision. An honest-but-incomplete spec
  beats a made-up one — a wrong number here poisons every downstream quote.
- Don't let the spec drift mid-interview; keep the numbers consistent.
- Collect only what's needed to price the job. No unnecessary personal data.
- Do not promise prices or vendors — you build the spec; the calls come later.

# TOOLS
- save_spec_field : upsert one field of the structured spec as you learn it.
- finalize_spec   : ONCE, after the client confirms the read-back. Freezes + hashes
                    the spec (spec_hash) so every call proves it quoted THIS job.
--------------------------------------------------------------------------------

================================================================================
 3. INTERVIEW MAP  (ask top-down; skip optional categories the client declines)
================================================================================
EVENT BASICS
  event.type (wedding|birthday|baby_shower), event.date (+ flexibility: weekday ok?
  daytime ok?), event.guest_count (+ how firm), location.city, location.region_profile,
  budget.total_ceiling, budget.allocation (rough split per category, if they have one)
VENUE      indoor/outdoor, seated/standing, hours needed, parking/AV, in-house catering?
CATERING   service_style (plated|buffet|family), dietary, alcohol, staff_required,
           equipment_included
DECOR      theme, palette, floral_style, table_setting, rentals
PHOTO(opt) hours of coverage, second shooter, deliverables
MUSIC(opt) dj|live_band, hours, PA/equipment
STYLE (weddings) palette, density, theme, floral_style, table_setting
Each maps to a dotted path in the spec JSON, e.g. catering.service_style,
event.date_flexible, budget.allocation.catering.

 4. TOOLS  (paste-ready; see ../tools/00_tool_config_reference.txt for the shape)
================================================================================
--- save_spec_field ---
{
  "tool_config": {
    "type": "webhook",
    "name": "save_spec_field",
    "description": "Save one field of the structured job spec as I learn it. Use a dotted path into the spec JSON. Set confidence to 'confirmed' when the client stated it clearly, 'assumed' when inferred, 'unknown' when they don't know yet.",
    "response_timeout_secs": 8,
    "api_schema": {
      "url": "{{server_base_url}}/api/agent-tools/spec/field",
      "method": "POST",
      "content_type": "application/json",
      "request_headers": { "Authorization": { "secret_id": "agent_tools_secret" } },
      "request_body_schema": {
        "type": "object",
        "properties": {
          "spec_draft_id": { "type": "string", "dynamic_variable": "spec_draft_id" },
          "path":          { "type": "string", "description": "dotted path, e.g. 'catering.service_style' or 'event.guest_count'" },
          "value":         { "type": "string", "description": "the value as stated by the client" },
          "confidence":    { "type": "string", "description": "confirmed | assumed | unknown" }
        },
        "required": ["spec_draft_id", "path", "value", "confidence"]
      }
    },
    "assignments": [
      { "source": "response", "dynamic_variable": "missing_required", "value_path": "missing_required", "preserve_native_type": true }
    ]
  }
}
Server returns: { "ok": true, "missing_required": ["venue.hours", "decor.theme"] }
-> the agent asks next whatever is still missing. May echo to the UI so the form fills
   live as the client talks.

--- finalize_spec ---
{
  "tool_config": {
    "type": "webhook",
    "name": "finalize_spec",
    "description": "Freeze the spec after the client confirms the full read-back. Call exactly once. If the server reports required fields still unknown, ask about them and do not finalize yet.",
    "response_timeout_secs": 8,
    "api_schema": {
      "url": "{{server_base_url}}/api/agent-tools/spec/finalize",
      "method": "POST",
      "content_type": "application/json",
      "request_headers": { "Authorization": { "secret_id": "agent_tools_secret" } },
      "request_body_schema": {
        "type": "object",
        "properties": {
          "spec_draft_id": { "type": "string", "dynamic_variable": "spec_draft_id" }
        },
        "required": ["spec_draft_id"]
      }
    },
    "assignments": [
      { "source": "response", "dynamic_variable": "spec_hash",    "value_path": "spec_hash" },
      { "source": "response", "dynamic_variable": "spec_summary", "value_path": "spec_summary" }
    ]
  }
}
Server: validate completeness, compute spec_hash, freeze. Reject (return the missing
fields) if any required field is still unknown.
Returns: { "ok": true, "spec_hash": "sha256:...", "spec_summary": "Wedding on ..." }

================================================================================
 5. DOCUMENT-INTAKE PATH  (the REQUIRED second intake path)
================================================================================
The same spec JSON can be produced from a photo / existing quote / inventory PDF via a
vision/OCR pass on the SERVER (not an ElevenLabs agent). Both paths MUST emit the
identical schema and go through the SAME finalize step / spec_hash contract, so a spec
built by voice and one built from a document are indistinguishable to the Caller.
Keep the schema defined in ONE place on the server and have both writers target it.
After a document parse, it's good practice to run this voice agent in "confirm mode":
pre-fill the fields, then only ask the client to confirm or correct — reuse
save_spec_field with confidence:"assumed" for parsed values.
